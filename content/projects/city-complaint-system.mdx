## Problem

Civic complaints — broken streetlights, water issues, garbage collection —
were reaching the city through phone calls and paper, with no tracking, no
ownership, and no way for a resident to know whether anything happened.
Response times suffered because the intake itself was the bottleneck.

## Approach

Build the simplest system that closes the loop: residents file a complaint
online with category and location, the system routes it to the right
department, and status updates are visible until resolution. I designed and
developed the platform end to end within the team's WordPress-based stack —
custom PHP for the complaint workflow, JavaScript for the interactive
front end.

## Architecture & Tech

WordPress as the CMS backbone with custom PHP modules for complaint
intake, categorization, routing, and status tracking, backed by MySQL.
The front end is hand-built HTML/CSS/JavaScript tuned for low-end devices
and patchy connectivity — the users are residents, not tech workers.

## Results & Impact

The system was shipped and used for real civic issue reporting, improving
issue visibility and response times over the manual process it replaced.
This is the project that taught me production software: real users, real
data, real consequences when a form breaks.

## What I'd do next

Today I'd rebuild it as a Next.js + Node application with a
PostgreSQL/Prisma backend, SMS notifications for status changes, and an
analytics view so the city can see *where* complaints cluster — the
architecture I now default to for exactly this kind of civic tool.
